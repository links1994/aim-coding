# Java Code Generation Skill

Generate Java microservice code conforming to project specifications based on technical specifications.

> **Mandatory Specifications**: Code generated by this Skill must strictly follow these specifications (in priority order):
> 1. **Coding Standards**: `.qoder/rules/04-coding-standards.md` - naming conventions, response formats, exception handling, etc.
> 2. **Architecture Standards**: `.qoder/rules/05-architecture-standards.md` - layered architecture, service responsibilities, interface styles, etc.
>
> **Must read above specification files before generating code** to ensure code conforms to project standards.

---

- Technical specification (design.md) generated and confirmed
- User explicit command: "generate code" or "delegate: generate code"

## Workflow

### Step 1: Read Configuration and Knowledge Base Query (Auto-reuse existing code)

1. **Read Input Documents**
   - Read technical specification (tech-spec.md)
   - Read API definition (openapi.yaml)
   - Read project resource mapping (RESOURCE-MAP.yml)
   - Read write scope (SCOPE.yml)
   - **Must read** coding standards and architecture standards

2. **Knowledge Base Query (Auto-reuse existing code)**

   **Use knowledge-base-query Skill to query**:
   ```
   Query type: feature
   Keywords: {current feature keywords}
   
   Purpose:
   - Query similar function code structures
   - Reuse existing Controller/Service/Mapper patterns
   - Reference existing DTO/VO definitions
   ```

   **Query types and purposes:**

   | Query Type | Purpose | Reuse Content |
   |------------|---------|---------------|
   | feature | Similar function archives | Controller/Service/Mapper code structures |
   | schema | Database table structures | Existing field definitions, avoid naming conflicts |
   | api | Internal/external APIs | Feign interface definitions, third-party API call patterns |
   | pitfall | Historical pitfalls | Avoid known traps (e.g., Feign client duplicate creation) |

   **Query result processing**:
   - If similar function has code → Reference its code structure, reuse reusable code snippets
   - If no similar function → Generate new code according to specifications

### Step 2: Service Analysis

Analyze each service:

- Determine service type (facade service / application service / support service)
- Determine code layers to generate (Controller/Service/Mapper/Entity/DTO/Feign)
- Determine dependency relationships (which services Feign calls)

### Step 3: Code Generation

Generate code by service in parallel, following specifications:

#### Object Type Specifications

| Object Type | Module | Package Location | Usage Scenario | Naming Rule |
|-------------|--------|------------------|----------------|-------------|
| **Remote Request** | API Module | `api/dto/request/` | Remote call request parameters | `XxxRequest` |
| **Remote Response** | API Module | `api/dto/response/` | Remote call response values | `XxxResponse` |
| **Feign Client** | API Module | `api/feign/` | Remote service call interface | `{ModuleName}RemoteService` |
| **Frontend Request** | Facade Service | `domain/dto/request/` | Frontend request parameters (local) | `XxxRequest` |
| **Frontend Response** | Facade Service | `domain/dto/response/` | Frontend response values (local) | `XxxResponse` / `XxxVO` |
| **Query** | Application Service | `domain/dto/` | Service layer query parameters | `XxxQuery` |
| **PageQuery** | Application Service | `domain/dto/` | Paged query parameters | `XxxPageQuery` |
| **ListQuery** | Application Service | `domain/dto/` | List query parameters | `XxxListQuery` |
| **DTO** | Application Service | `domain/dto/` | Internal data transfer | `XxxDTO` |
| **DO** | Application Service | `domain/entity/` | Database entity | `AimXxxDO` |

**Mandatory Constraints**:

- **Controller Layer**: Request parameters must end with `Request`, response parameters must end with `Response` or `VO`
- **Service Layer Internal**: Request parameters end with `Query`, response ends with `DTO`
- **Prohibited Mixing**: Controller layer must not use `Query` or `DTO` as parameter/return type suffixes

**Layer Responsibilities**:

- **API Module** (mall-{module}-api): Stores remote call interfaces, referenced by other services
- **Facade Service**: Frontend Request/Response placed locally in `domain/dto/`, not exposed externally
- **Application Service**: Internal DTOs/Queries/DOs not exposed externally, remote interfaces exposed through API module
- Request/Response **only used in Controller layer**, not directly passed to Service layer
- Controller layer responsible for conversion: Request → Query/DTO → Call Service
- Service layer parameters can only be Query or DTO, cannot be Request

**Query Parameters vs Update Request Naming Distinction**:

| Scenario | Naming Suffix | Usage Location | Example |
|----------|---------------|----------------|---------|
| **Paged Query** | `PageQuery` | Service layer internal | `JobTypePageQuery` |
| **List Query** | `ListQuery` | Service layer internal | `JobTypeListQuery` |
| **Partial Update** | `Request` | Controller layer | `JobTypeStatusRequest` |

**Key Distinction**:
- `PageQuery/ListQuery`: Used for query scenarios, containing query conditions + pagination info, used internally in Service layer
- `Request`: Used for update scenarios (especially partial field updates), Controller layer receives frontend requests

**Facade Service Calling Remote Service Naming Distinction**:

| Object Type | Facade Service (Local) | Remote Service (API Module) | Naming Example |
|-------------|------------------------|----------------------------|----------------|
| Request Parameter | `XxxRequest` | `XxxApiRequest` | Local: `JobTypeCreateRequest`, Remote: `JobTypeCreateApiRequest` |
| Response | `XxxResponse` | `XxxApiResponse` | Local: `JobTypeResponse`, Remote: `JobTypeApiResponse` |

**Naming Strategy**:

- Remote service objects uniformly add `Api` suffix (e.g., `JobTypeCreateApiRequest`)
- Local service objects keep standard naming (e.g., `JobTypeCreateRequest`)
- VO only used in complex object aggregation scenarios, not as differentiation means

#### Query Return Object Specifications

##### Application Service (mall-agent/mall-user)

| Layer | Return Type | Description |
|-------|-------------|-------------|
| **Mapper** | `AimXxxDO` / `AimXxxDTO` | Raw data objects |
| **Service** | `CommonResult<CommonResult.PageData<XxxResponse>>` | Business logic + DO to Response conversion |
| **InnerController** | `CommonResult<CommonResult.PageData<XxxResponse>>` | Directly forward Service results |

**Application Service Constraints**:

- **Only return Response**, do not wrap VO
- **Parameter Rules**: ≤2 primitive types use `@RequestParam`, otherwise always `@RequestBody`
- **Service Layer**: Complete DO to Response conversion

##### Facade Service (mall-admin/mall-app)

| Scenario | Return Type | Description |
|----------|-------------|-------------|
| **Simple Forward** | `CommonResult<CommonResult.PageData<XxxResponse>>` | Directly return single Feign result |
| **Data Aggregation** | `CommonResult<XxxVO>` | VO wraps multiple Responses |

**Facade Service Constraints**:

- **Allow VO usage**: When needing to aggregate multiple application service data
- **Parameter Rules**: GET can use multiple `@RequestParam`, POST uses `@RequestBody`
- **No Service Layer**: Controller directly calls Feign

#### Add/Update/Delete Return Specifications

**Application Service Service Layer**:

| Operation Type | Service Return | InnerController Return | Description |
|----------------|----------------|------------------------|-------------|
| **Create** | `Long` (new ID) / `boolean` | `CommonResult<Long>` / `CommonResult<Void>` | Return new record ID or success flag |
| **Update** | `boolean` / `int` (affected rows) | `CommonResult<Void>` | Return success or affected rows |
| **Delete** | `boolean` / `int` (affected rows) | `CommonResult<Void>` | Return success or affected rows |

**Specification Notes**:

- **Do not return complete objects**: Create/Update operations do not return complete entity objects
- **Return operation results**: Return ID, affected rows, or success flag
- **Query and modification separation**: When needing updated data, call query method separately
- **InnerController Responsibility**: Convert Service operation results to Response

**VO Usage Example** (Facade Service only):

```java
// VO aggregates multiple local Responses (converted from ApiResponse)
@Data
public class AgentDetailVO implements Serializable {
    private AgentResponse agent;           // Local Response (converted from AgentApiResponse)
    private UserResponse creator;          // Local Response (converted from UserApiResponse)
    private List<SkillResponse> skills;    // Local Response (converted from SkillApiResponse)
}

// Facade Service Controller
@GetMapping("/{agentId}/detail")
public CommonResult<AgentDetailVO> getAgentDetail(@PathVariable("agentId") Long agentId) {
    // Call multiple RemoteServices to get remote ApiResponses
    AgentApiResponse agentApi = agentRemoteService.getById(agentId).getData();
    UserApiResponse creatorApi = userRemoteService.getById(agentApi.getCreatorId()).getData();

    // Convert to local Response
    AgentResponse agent = convertToAgentResponse(agentApi);
    UserResponse creator = convertToUserResponse(creatorApi);

    // Assemble VO
    AgentDetailVO vo = new AgentDetailVO();
    vo.setAgent(agent);
    vo.setCreator(creator);

    return CommonResult.success(vo);
}
```

#### Interface Styles

| Service Type | Style | Path Prefix |
|--------------|-------|-------------|
| Facade Service | RESTful | `/admin/api/v1/` or `/app/api/v1/` |
| Application Service | Simplified | `/inner/api/v1/` |

#### Pagination Scheme Selection

**Interface design must estimate data volume**:

| Data Volume | Pagination Scheme | QueryService Call | AimXxxService Implementation | Description |
|-------------|-------------------|-------------------|------------------------------|-------------|
| **Small Table** (< 1M) | MyBatis-Plus Pagination | `aimXxxService.page()` | MP `page()` method | Simple and efficient |
| **Large Table** (>= 1M) | Covering Index Pagination | `aimXxxService.pageByCoveringIndex()` | Call `AimXxxMapper.selectIdsByPage()` + `selectBatchByIds()` | Avoid table lookup, has total |

**Layered Call Principles**:

- **XxxQueryService** can only call `AimXxxService` methods
- **AimXxxService** encapsulates pagination logic, internally can call `AimXxxMapper`
- **AimXxxMapper** only exposed to `AimXxxService`

**Example**:

```java
/**
 * Order list
 *
 * Data volume estimate: Expected > 10M records (large table)
 * Pagination scheme: Covering index pagination
 */
@GetMapping("/orders")
public CommonResult<CommonResult.PageData<OrderResponse>> pageOrder(
        @RequestParam(defaultValue = "1") Integer pageNum,
        @RequestParam(defaultValue = "20") Integer pageSize) {
    return orderDomainService.pageOrder(pageNum, pageSize);
}

// OrderQueryService
@Service
@RequiredArgsConstructor
public class OrderQueryService {
    private final AimOrderService aimOrderService;

    public Page<AimOrderDO> pageOrder(Integer pageNum, Integer pageSize) {
        // Can only call AimOrderService, not directly call AimOrderMapper
        return aimOrderService.pageByCoveringIndex(pageNum, pageSize);
    }
}

// AimOrderService (extends IService<AimOrderDO>)
@Service
public class AimOrderService extends ServiceImpl<AimOrderMapper, AimOrderDO> {

    public Page<AimOrderDO> pageByCoveringIndex(Integer pageNum, Integer pageSize) {
        // Internally call AimOrderMapper
        int offset = (pageNum - 1) * pageSize;
        List<Long> ids = baseMapper.selectIdsByPage(offset, pageSize);
        // ...
    }
}
```

## SQL Writing Specifications

### XML Mapper Priority

- **SQL uniformly placed in XML**: `src/main/resources/mapper/XxxMapper.xml`
- **Annotations only for simple queries**: Single table queries, no dynamic conditions, SQL not exceeding one line
- **Prohibit complex SQL in annotations**: SQL containing `<script>`, `<if>`, `<foreach>` and other dynamic tags

### XML Specification

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.aim.mall.agent.mapper.JobTypeMapper">

    <!-- Extract repeated SQL -->
    <sql id="Base_Column_List">
        id, code, name, status, sort_order, description, is_default, create_time, update_time
    </sql>

    <sql id="Base_Table">
        aim_job_type
    </sql>

    <sql id="Base_Where">
        WHERE is_deleted = 0
    </sql>

    <!-- Paged query -->
    <select id="selectPageByKeyword" resultType="com.aim.mall.agent.domain.entity.AimJobTypeDO">
        SELECT
        <include refid="Base_Column_List"/>
        FROM
        <include refid="Base_Table"/>
        <include refid="Base_Where"/>
        <if test="keyword != null and keyword != ''">
            AND (name LIKE CONCAT('%', #{keyword}, '%') OR description LIKE CONCAT('%', #{keyword}, '%'))
        </if>
        ORDER BY sort_order ASC, create_time DESC
        LIMIT #{limit} OFFSET #{offset}
    </select>

</mapper>
```

**Specification Points**:

1. **Must extract repeated SQL**: Use `<sql>` tags to define column names, table names, common conditions
2. **Prohibit `SELECT *`**: Explicitly list all fields
3. **Prohibit annotation dynamic SQL**: Complex dynamic conditions must be written in XML
