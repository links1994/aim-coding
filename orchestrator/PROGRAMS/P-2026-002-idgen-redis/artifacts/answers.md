# 需求澄清确认记录

## 问题 1：日期格式支持

**确认结果**：至少支持以下 4 种日期格式

| 格式 | 示例 | 说明 |
|------|------|------|
| `yyyyMMdd` | 20260302 | 年月日 |
| `yyyyMMddHHmmss` | 20260302153045 | 年月日时分秒 |
| `yyyyMM` | 202603 | 年月 |
| `yyyy` | 2026 | 年 |

## 问题 2：序号位数

**确认结果**：
- 自增序号固定为 6 位，不足补零（示例：`000001`, `000010`, `999999`）
- 业务前缀支持 1-3 位大写字母（示例：`OR`, `EMP`, `A`）

## 问题 3：序号溢出处理

**初始确认**：当序号超过 6 位最大值（999999）时，重置并从 1 开始
- 注意：此方式在极端高并发场景下可能产生重复编码，需业务方注意

**后续澄清（重要变更）**：
- 重置会导致重复编码问题（如唯一键冲突）
- **最终方案**：达到上限时报错，不自动重置
- 错误码：`40002002`，错误信息：`当日序号已用完`
- 建议：高并发业务使用 `yyyyMMddHHmmss` 格式（按秒），几乎不可能用完

## 问题 4：Redis Key 前缀

**初始确认**：Redis Key 格式为 `{业务前缀}:{日期}`
- 示例：`OR:20260302`, `EM:202603`

**后续澄清（技术方案变更）**：
- 采用号段模式（MySQL + Redis）
- Redis Key 格式变更为：`idgen:segment:{prefix}:{datePattern}:{dateValue}`
- 示例：`idgen:segment:OR:yyyyMMdd:20260302`
- Redis 数据结构：Hash（包含 current/max/step 字段）

## 问题 5：Redis 闪断问题

**问题**：如果 Redis 出现闪断，数据会丢失，导致序号重复

**澄清结果**：
- 采用号段模式，MySQL 持久化号段分配
- Redis 闪断后，可从 MySQL 重新加载号段
- 数据安全得到保障

## 问题 6：技术方案选择

**讨论方案**：
- A. 纯 Redis（简单但数据不安全）
- B. Redis + MySQL 双写（写操作多，性能差）
- C. 纯 MySQL（简单但性能一般）
- D. 号段模式（MySQL 分配号段 + Redis 缓存自增）✅ **选中**

**选中方案 D（号段模式）理由**：
- 平时性能极高（Redis 内存操作）
- 数据安全（MySQL 持久化）
- 数据库压力小（批量取号）
- 业界常用方案（美团 Leaf、滴滴 TinyID）

## 问题 7：号段用尽处理

**问题**：如果号段达到最大值（999999）怎么办？

**澄清结果**：
- 号段用尽时从 MySQL 申请新号段
- 当序号达到 999999 时，报错不重置
- 避免产生重复编码

## 问题 8：数据初始化策略

**问题**：初次调用时，数据是自动初始化还是手动配置？

**澄清结果**：
- **采用自动初始化**
- 首次调用时自动检测 MySQL 记录是否存在
- 不存在则自动插入默认配置（step_size=1000, current_max_seq=0）
- 业务方无感知，无需手动配置
